# Toolpack SDK v2.3.0

Date: 2026-06-13

## Highlights

- **Mode race fix** — concurrent agents sharing a Toolpack instance no longer corrupt each other's tool filtering, system prompt, or response format; mode is now snapshotted per-request
- **Per-request mode override** — callers can pin a specific mode for a single `generate()`/`stream()` call without touching instance state
- **`ToolpackInitConfig` on `CompletionRequest`** — full init config is now the public type for request-level mode configuration

None.

## Bug fixes

### Mode race condition in shared Toolpack instances

**Problem:** `activeMode` is mutable instance state on `AIClient`. In multi-agent setups where several agents share one Toolpack instance (e.g. via `AgentRegistry`), a concurrent `setMode()` call — such as the one made by a delegated sub-agent partway through a parent agent's tool loop — could change the active mode between rounds of the parent's request. The wrong tools, system prompt, and response format would be applied to later rounds.

**Fix:** `AIClient.generate()` and `AIClient.stream()` now snapshot the effective mode at the very start of each request and pass that snapshot explicitly through every internal call (`enrichRequestWithTools`, `injectModeSystemPrompt`, `injectBaseAgentContext`, `executeTool`, `executeToolSearch`). A `setMode()` from any source after the snapshot is taken has no effect on that request.

The snapshot is taken from `request.mode` when present, otherwise from `this.activeMode` at call time. This means mode selection is now deterministic and isolated per request.

## New features

### Per-request mode on `CompletionRequest`

```typescript
// Pin a mode for a single call — instance activeMode is not changed
const result = await toolpack.generate({
  messages: [...],
  mode: 'publisher',       // string name (resolved from registry)
});

// Or pass a ModeConfig directly
const result = await toolpack.generate({
  messages: [...],
  mode: myModeConfig,
});

// Explicitly run without any mode for this request
const result = await toolpack.generate({
  messages: [...],
  mode: null,
});
```

The new `mode` field on `CompletionRequest` accepts:
- `ModeConfig` — use this config directly
- `string` — resolved from Toolpack's mode registry (throws for unknown names)
- `null` — run with no mode for this request
- omitted — snapshot the instance's active mode at request start (existing behavior)

### `resolveRequestModeName()` in Toolpack facade

String mode names on `request.mode` are resolved to `ModeConfig` by `Toolpack.generate()` and `Toolpack.stream()` before the request reaches `AIClient`. This keeps the mode registry in `Toolpack` (its owner) and away from `AIClient`.

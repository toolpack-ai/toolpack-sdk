# Toolpack SDK v2.5.0

Date: 2026-07-12

## Highlights

- **Skills: system prompt injection** — skill instructions are now appended to the system prompt instead of the user message, where behavioral rules are followed more reliably
- **Skills: multi-turn query** — BM25 search now uses the last 3 user messages so short follow-ups like "performance?" still match the right skill
- **Skills: camelCase tokenizer fix** — `codeReview` now correctly splits into `code` + `review` for BM25 indexing
- **Skills: YAML block-list tag parsing** — tags written as YAML block lists (`- tag` on separate lines) now parse correctly
- **Skills: larger instructions limit** — raised from 2,000 to 5,000 characters
- **Skills: trigger count warning** — validator now warns when a skill has fewer than 3 triggers

---

## Breaking changes

**Skill instructions injection target changed.** Instructions are now appended to the system prompt (or a new system message is created at position 0 if none exists) instead of being prepended to the last user message. If you have tooling that inspects or parses the injected user message content, update it to look at the system message instead.

This change makes skill instructions follow more reliably — the model treats system prompt content as rules and user message content as requests. The injected format (`<skill-instructions>...</skill-instructions>`) is unchanged.

## Bug fixes

### camelCase tokenizer dead code (BM25)

`tokenize()` called `toLowerCase()` before the camelCase split regex `([a-z])([A-Z])`. Since all letters were already lowercase by the time the regex ran, it never matched — `codeReview` was indexed as the single unsearchable token `codereview`.

**Before:** `codeReview` → `codereview` (no match on `code` or `review`)  
**After:** `codeReview` → `code review` (searchable by either word)

The fix swaps the order: camelCase split first, then lowercase.

---

### YAML block-list tag parsing

The tag parser only handled inline array syntax (`tags: ["a", "b"]`). YAML block-list syntax was silently broken:

```yaml
tags:
  - coding
  - quality
```

The `- coding` lines have no colon, so the parser skipped them and returned an empty tag list. Fixed by adding `parsingTagsList` state tracking that collects subsequent `  - value` lines after a bare `tags:` key.

## Improvements

### Skill instructions injected into system prompt

`createSkillInterceptor` previously prepended `<skill-instructions>` to the last user message. It now appends to the system prompt instead — or creates a new `{ role: 'system' }` message at position 0 if the request has none.

Fallback: if the system message has non-string content (rare), injection falls back to the user message to avoid a runtime error.

---

### BM25 query uses last 3 user messages

Previously the interceptor queried BM25 with only the last user message. A short follow-up like `"performance?"` on its own scores poorly and may not reach `minScore`. The query is now built by concatenating the last 3 user messages, giving each follow-up the context of the preceding turns.

---

### Validator warns on low trigger count

Skills with fewer than 3 triggers now produce a non-fatal validation warning at startup:

```
only 1 trigger(s) defined; add more variations (synonyms, short forms, longer phrases)
so BM25 can match a wider range of user phrasings — aim for at least 3
```

The existing error for zero triggers is unchanged.

---

### Instructions limit raised to 5,000 characters

`SKILL_LIMITS.instructions` increased from 2,000 to 5,000 characters. The worst-case context footprint with default settings is now `3 skills × 5,000 chars = 15,000 chars`.

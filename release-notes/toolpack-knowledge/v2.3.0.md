# @toolpack-sdk/knowledge v2.3.0

Date: 2026-06-13

## Highlights

- **`reSync` data-loss bug fixed** — setting `reSync: false` on either `Knowledge.create()` options or `PersistentKnowledgeProvider` options is now sufficient; previously both had to be set or runtime-added chunks were silently wiped on every restart

## Breaking changes

None.

## Bug fixes

### `reSync: false` had to be set in two places to preserve runtime-added chunks

**Problem:** `Knowledge.create()` consulted `provider.shouldReSync()` only when `options.reSync !== false`. If a caller followed the documented pattern of setting `reSync: false` only on the Knowledge level (the intuitive place), `shouldReSync()` was called with no argument, fell back to `this.options.reSync` (which was `undefined`), and returned `true` — triggering a full `sync()` whose first step is `provider.clear()`. Every process restart silently wiped all chunks added at runtime via `knowledge.add()`.

The same bug existed in reverse: setting `reSync: false` only on the provider (as shown in the package README's first example) also failed, because `create()` never consulted `shouldReSync()` unless the Knowledge-level flag was already `false`.

**Fix:** `create()` now consults `shouldReSync()` whenever the provider exposes it, passing `options.reSync` as the caller's intent. `shouldReSync(callerReSync?)` resolves the effective flag as `providerOption ?? callerReSync` — the provider's explicit setting wins when present, otherwise the Knowledge-level flag applies. A single `reSync: false` in either place is now sufficient.

**Migration:** No code changes required. Callers that already set the flag in both places continue to work identically. Callers that only set it in one place now get the behavior they expected.

```typescript
// Any of these now correctly preserves runtime-added chunks across restarts:

// 1. Knowledge-level only
await Knowledge.create({ provider, sources, embedder, description, reSync: false });

// 2. Provider-level only (README example)
const provider = new PersistentKnowledgeProvider({ namespace: 'my-kb', reSync: false });
await Knowledge.create({ provider, sources, embedder, description });

// 3. Both (unchanged — provider wins if they conflict)
const provider = new PersistentKnowledgeProvider({ namespace: 'my-kb', reSync: false });
await Knowledge.create({ provider, sources, embedder, description, reSync: false });
```

**First-run behavior unchanged:** when the store is empty, `shouldReSync()` still returns `true` regardless of the flag — so the initial indexing from `sources` always runs.

## API changes

### `PersistentKnowledgeProvider.shouldReSync(callerReSync?: boolean)`

The signature of the internal `shouldReSync()` method gains an optional parameter. This is not part of the public `KnowledgeProvider` interface — it is called only by `Knowledge.create()`. No consumer should be calling this directly, but the signature change is documented for completeness.

### `KnowledgeOptions.reSync` — JSDoc clarified

The `reSync` field now has a full doc comment explaining the clear/wipe semantics and confirming that one flag in either place is sufficient.
